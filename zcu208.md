# Platform: ZCU208

## Device
- Board: AMD Zynq UltraScale+ RFSoC ZCU208 Evaluation Kit
- FPGA/SoC: XCZU48DR-2FSVG1517E (RFSoC Gen 3)
- Process: TSMC 16nm FinFET+

---

## ADC Specifications
- Channels: 8× ADC
- Resolution: **14-bit**
- Sample rate: **5 GSPS per channel**
- Nyquist frequency: 2.5 GHz
- **Analog input bandwidth: 6 GHz (-3 dB)** — see sampling-theory.md
- Architecture: Real ADC. IQ output is synthesised digitally via on-chip DDC.
  Do NOT treat ZCU208 IQ output as implying a 2.5 GHz carrier ceiling.

## DAC Specifications
- Channels: 8× DAC
- Resolution: 14-bit
- Sample rate: 10 GSPS per channel
- Output bandwidth: up to ~5 GHz usable

---

## Key Advantages for Radar
- 14-bit resolution — best available at this sample rate from any vendor
- 8 ADC + 8 DAC channels maps exactly to 8 TX + 8 RX phased array requirement
- Multi-Tile Synchronisation (MTS) — all 8 ADC and 8 DAC channels are
  phase-coherent with each other. Essential for beamforming.
- Monolithic integration eliminates PCB interface noise between ADC and
  digital fabric — key reason 14-bit is achievable at 5 GSPS

## Add-On Cards (all included in kit)
| Card | Purpose |
|---|---|
| **XM655** | RF breakout card. Exposes ADC/DAC ports as SMA connectors with baluns. Primary interface for custom RF chains. No external balun design needed. |
| **XM650** | N79 band loopback card. Used for out-of-box evaluation only. |
| **CLK104** | RF clock add-on. Provides low-jitter reference clock distribution. Accepts external reference for LO-synchronised operation. Critical for coherent radar. |

**Both XM655 and CLK104 are required for radar operation.** XM650 is not needed.

### XM655 Balun Frequency Ranges
The XM655 has two sets of baluns covering different frequency ranges.
Verify which balun connections to use on the board for your chosen IF frequency.
ADI documentation refers to "low frequency" and "high frequency" balun connections.

## Host Interface
- 2× SFP28 (4 ports total, 4× 25G = 100G aggregate)

## DDC Configuration

The DDC (Digital Down Converter) in the ZU48DR is **hardened IP** — a fixed
hardware block in the silicon, not soft logic implemented in LUTs. AMD calls
this the RF Data Converter IP.

### What can be configured at runtime (via PYNQ, no bitstream rebuild needed)
- NCO frequency per channel (sets the digital downconversion centre frequency)
- Decimation ratio
- Mixer gain
- Real vs IQ output mode per channel

These are register writes to the hardened block, exposed as Python API calls
in PYNQ. Example: `adc_tile.ADCChannel[0].MixerSettings`

### What requires a bitstream rebuild (Vivado, license included in kit)
- Changing the fundamental DDC architecture
- Changing how many DDC chains are active
- Changing data path width or tile configuration

### Practical implication
For prototyping, PYNQ handles all DDC configuration. The toolchain concern
only arises if the fundamental FPGA design needs to change. The DDC is not
"software you write" — it's hardened silicon you configure via registers.

## mmWave Note
"Extended mmWave support" in AMD marketing refers to the **RFMC 2.0 connector**
which accepts external daughter cards (e.g. Otava DTRX2 for 20–30 GHz).
This is NOT native ADC capability. The ZU48DR ADC cannot directly sample
above 6 GHz regardless of what daughter cards are attached.
